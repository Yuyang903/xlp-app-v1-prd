享乐派App_v1.0 产品需求文档 (PRD)

产品名称：享乐派

文档版本：v1.0

编写日期：2026-03-30

1. 产品概述

享乐派App_v1.0 是一款专注于音乐实体专辑数字化体验与粉丝互动的移动端应用。
用户可以通过 NFC 或扫描二维码将购买的实体专辑绑定至 App，获取专属的数字专辑内容（包括音乐、MV、电子写真、数字小卡、专属徽章等），
并通过权益机制参与互动活动（如赠品领取资格、线下活动抽奖等）。

2. 核心概念与资产关联关系

享乐派 App_v1.0 围绕实体介质与数字资产的映射展开，以下是系统中的核心概念及其关联逻辑：

2.1 核心概念

  • NFC卡 / 二维码：绑定数字资产的物理介质。无论是内置在实体专辑、小卡中的 NFC 芯片，还是印刷在商品上的二维码，其底层原理一致，唯一性NFC均具有系统级的全局唯一性，作为用户在线下确认身份并领取对应数字资产的唯一密钥。公用NFC卡不具有唯一性，任何人感应/扫码均可以用于绑定徽章或小卡。
  • 商品 (SKU)：用户实际购买的物理或虚拟产品配置（例如：A平台特典版实体专辑、B平台常规版实体专辑、独立小卡盲盒等）。
  • 收藏品：数字资产的顶层容器。对应某个具体的音乐专辑或项目，包含该专辑下的所有音乐、MV、写真、社区权限以及专属的小卡图鉴和徽章墙。（注：多个商品配置可能映射到同一个收藏品，例如不同平台的特典版实体专辑，激活后获得的是相同的核心数字收藏品内容）。
  • 数字资产：
    • 音乐：收藏品内的核心内容资产，通常为无损音质的音频资源，用户解锁收藏品且拥有数字资产权限后即可在 App 内收听。
    • MV：收藏品内的视频资产，包含官方音乐视频、VLOG、拍摄花絮等专属影像内容。
    • 写真：收藏品内的图像资产，为专属艺人的高清电子写真图集或海报。
  • 增值资产：
    • 数字小卡：隶属于特定“收藏品”下的增值卡片资产，收集小卡可点亮对应的图鉴，可链接关联线上活动。（注：普通数字小卡不可重复获取，即用户图鉴中每种普通小卡最多仅展示一张）。
    • 编码数字小卡（稀有玩法）：数字小卡的一种高阶稀有形态。在特定商品或活动中发行，每张编码小卡除了拥有常规数字小卡的属性外，还附带一个全服唯一的独立编号（如 NO. 0089757）。编号的唯一性赋予了该卡片极高的收藏价值与稀缺性，未来可作为核心粉丝身份的象征，或用于解锁特定的顶级权益（如优先购票、限量周边兑换等）。（注：由于具有唯一编号，即使是卡面相同的编码数字小卡，只要编号不同，用户即可重复获取并在图鉴中同时拥有多张）。
    • 徽章：荣誉与打卡凭证。用户完成特定任务（如线下NFC打卡、参与线上活动等）后获得的数字标识。特定的徽章或徽章组（如集齐7个子徽章合成完整徽章）可作为解锁高级权益的凭证。
  • 数字资产权限：收藏品是数字资产的容器，同时也是小卡、徽章的容器，但拥有收藏品不代表必定拥有数字资产的消费权限。例如，在某些特定场景下发放的独立电子徽章或小卡，用户激活后仅会点亮该收藏品容器及对应的增值资产，但无法激活核心数字资产权限，此时用户将无法收听该专辑的音乐或播放专属 MV。
  • 权益：附加福利与特权。用户持有特定的数字资产（如收藏品、数字小卡、徽章等）后所解锁的专属福利资格，包含但不限于实物赠品领取、线下活动抽奖、优先购买权及专属内容访问权限等。

2.2 NFC 的链接种类

根据使用场景和绑定逻辑的不同，NFC 芯片在系统中分为以下两种：

  • 唯一性 NFC：芯片具有强身份属性，具有系统级全局唯一性。App 感应后可与用户账号进行唯一绑定，激活后对应的物理介质将与该用户账号强绑定，不可被其他账号再次激活。通常用于新版专辑配置的标准卡、独立购买的实体小卡等。
  • 公用 NFC：芯片不与单一用户绑定，支持被多个不同的用户重复感应识别。通常用于线下打卡活动（如演唱会场馆、快闪店等固定打卡点），用户打卡后可获取特定的“数字小卡”、“专属徽章”或跳转特定活动链接。公用 NFC 可配合“徽章组合”扩展玩法使用。

2.3 小卡获取方式

  • 默认发放：当用户激活绑定了特定收藏品的 NFC 卡时，若该专辑系统配置有“默认小卡”（例如全员普赠的基础卡），用户在获得收藏品的同时，将自动同步获取这些默认小卡。
  • 感应获取：用户通过识别带有唯一绑定属性的实体小卡盲盒或特典卡，激活后获取与该物理介质一一对应的某一张特定数字小卡。普通数字小卡在用户图鉴中不可重复获取。
  • 感应获取-编码版：用户感应获取的是“编码数字小卡”。由于其具有唯一编号（如 NO. 0089757），即使用户感应多张卡面相同的编码小卡，只要编号不同，即可重复获取并在图鉴中同时拥有多张。

2.4 资产绑定路径

物理介质（NFC卡）与数字资产的映射与激活路径分为以下几种情况：

  1. NFC 绑收藏品
    • 路径：NFC卡 → 商品 → 收藏品
    • 说明：用户激活后，直接获得该商品对应的“数字收藏品（完整版）”，并拥有完整的数字资产权限（可收听音乐、观看 MV 等）。
  2. NFC 绑小卡（同时拥有该收藏品）（有数字资产权限）
    • 路径：NFC卡 → 商品 → 收藏品 → 特定小卡
    • 说明：激活后，用户在点亮特定数字小卡的同时，也获得了该小卡所属的完整“数字收藏品”，并且拥有该收藏品下核心数字资产的消费权限（可听歌、看 MV 等）。
  3. NFC 绑小卡（同时拥有该收藏品）（无数字资产权限）
    • 路径：NFC卡 → 商品 → 收藏品容器 → 特定小卡
    • 说明：在特定活动或独立小卡售卖场景中，激活后用户获得了特定数字小卡，并点亮了该小卡所属的“收藏品”容器，但未获得该收藏品下核心数字资产的权限。用户可以在图鉴中欣赏小卡，但无法在该收藏品下收听音乐或观看 MV。
  4. NFC 绑徽章（同时拥有该收藏品）（无数字资产权限）
    • 路径：NFC卡 → 活动/商品 → 收藏品容器 → 特定徽章
    • 说明：常见于线下打卡等场景，用户感应后获得了特定的荣誉徽章，并点亮了对应的“收藏品”容器，但同样未获得该收藏品下核心数字资产的权限（无法听歌/看MV）。

3. 核心资产数据库结构设计

为了支撑上述的小卡、NFC、收藏品、徽章及复杂的权限绑定关系，系统需要设计合理的底层数据表结构。以下为核心表结构设计（点击展开查看详细字段）：

点击展开/折叠：核心资产数据库表结构

1. `nfc_tags` (NFC介质/二维码表)

记录发行的所有物理介质（NFC或二维码）及其类型与激活状态。

CREATE TABLE `nfc_tags` (
  `id` BIGINT AUTO_INCREMENT PRIMARY KEY,
  `uid` VARCHAR(64) UNIQUE NOT NULL COMMENT 'NFC芯片唯一UID或二维码内容',
  `type` TINYINT NOT NULL COMMENT '介质类型: 1-唯一性NFC(绑定即消耗), 2-公用NFC(可重复打卡)',
  `target_type` VARCHAR(32) NOT NULL COMMENT '绑定目标类型: collection(藏品), card(小卡), badge(徽章)',
  `target_id` BIGINT NOT NULL COMMENT '关联的目标ID(具体商品ID或活动ID)',
  `is_activated` TINYINT DEFAULT 0 COMMENT '激活状态: 0-未激活, 1-已激活(针对唯一性NFC)',
  `activated_by` BIGINT DEFAULT NULL COMMENT '激活该NFC的用户ID',
  `activated_at` DATETIME DEFAULT NULL COMMENT '激活时间',
  `created_at` DATETIME DEFAULT CURRENT_TIMESTAMP
);

2. `collections` (数字收藏品字典表)

定义系统中的顶级数字专辑容器。

CREATE TABLE `collections` (
  `id` BIGINT AUTO_INCREMENT PRIMARY KEY,
  `name` VARCHAR(128) NOT NULL COMMENT '收藏品名称(如: STUPID POP SONGS)',
  `artist` VARCHAR(64) NOT NULL COMMENT '艺人名称',
  `cover_url` VARCHAR(255) COMMENT '封面图URL',
  `description` TEXT COMMENT '专辑介绍',
  `created_at` DATETIME DEFAULT CURRENT_TIMESTAMP
);

3. `cards_dict` (数字小卡字典表)

定义所有可用的小卡模板，包括普通卡和编码卡。

CREATE TABLE `cards_dict` (
  `id` BIGINT AUTO_INCREMENT PRIMARY KEY,
  `collection_id` BIGINT NOT NULL COMMENT '所属收藏品ID',
  `name` VARCHAR(64) NOT NULL COMMENT '小卡名称',
  `image_url` VARCHAR(255) NOT NULL COMMENT '小卡图片',
  `is_coded` TINYINT DEFAULT 0 COMMENT '是否为编码稀有卡: 0-否, 1-是',
  `total_issue` INT DEFAULT 0 COMMENT '发行总数(针对编码卡)',
  `created_at` DATETIME DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (`collection_id`) REFERENCES `collections`(`id`)
);

4. `badges_dict` (徽章字典表)

定义系统中可获得的各种荣誉徽章。

CREATE TABLE `badges_dict` (
  `id` BIGINT AUTO_INCREMENT PRIMARY KEY,
  `collection_id` BIGINT NOT NULL COMMENT '所属收藏品容器ID',
  `name` VARCHAR(64) NOT NULL COMMENT '徽章名称(如: 现场打卡徽章)',
  `icon_url` VARCHAR(255) NOT NULL COMMENT '徽章图标URL',
  `created_at` DATETIME DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (`collection_id`) REFERENCES `collections`(`id`)
);

5. `user_collections` (用户收藏品资产表)

记录用户拥有的收藏品容器及核心数字资产权限。

CREATE TABLE `user_collections` (
  `id` BIGINT AUTO_INCREMENT PRIMARY KEY,
  `user_id` BIGINT NOT NULL COMMENT '用户ID',
  `collection_id` BIGINT NOT NULL COMMENT '收藏品ID',
  `has_digital_rights` TINYINT DEFAULT 0 COMMENT '是否有数字资产权限(听歌/看MV): 0-无, 1-有',
  `source_nfc_id` BIGINT COMMENT '来源NFC ID',
  `created_at` DATETIME DEFAULT CURRENT_TIMESTAMP,
  UNIQUE KEY `idx_user_collection` (`user_id`, `collection_id`),
  FOREIGN KEY (`collection_id`) REFERENCES `collections`(`id`)
);

6. `user_cards` (用户小卡资产表)

记录用户具体拥有的小卡实例。普通卡去重，编码卡可重复并记录唯一编号。

CREATE TABLE `user_cards` (
  `id` BIGINT AUTO_INCREMENT PRIMARY KEY,
  `user_id` BIGINT NOT NULL COMMENT '用户ID',
  `card_dict_id` BIGINT NOT NULL COMMENT '小卡字典ID',
  `collection_id` BIGINT NOT NULL COMMENT '冗余:所属收藏品ID',
  `serial_number` VARCHAR(32) DEFAULT NULL COMMENT '唯一编码(仅编码小卡有值,如: NO.038495)',
  `source_nfc_id` BIGINT COMMENT '来源NFC ID',
  `created_at` DATETIME DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (`card_dict_id`) REFERENCES `cards_dict`(`id`)
);

7. `user_badges` (用户徽章资产表)

记录用户点亮的徽章记录。

CREATE TABLE `user_badges` (
  `id` BIGINT AUTO_INCREMENT PRIMARY KEY,
  `user_id` BIGINT NOT NULL COMMENT '用户ID',
  `badge_dict_id` BIGINT NOT NULL COMMENT '徽章字典ID',
  `collection_id` BIGINT NOT NULL COMMENT '冗余:所属收藏品ID',
  `source_nfc_id` BIGINT COMMENT '打卡来源NFC ID',
  `created_at` DATETIME DEFAULT CURRENT_TIMESTAMP,
  UNIQUE KEY `idx_user_badge` (`user_id`, `badge_dict_id`),
  FOREIGN KEY (`badge_dict_id`) REFERENCES `badges_dict`(`id`)
);

4. 用户体系

  • 账号体系：是用户在 App 内的唯一身份标识，所有的数字资产（专辑、小卡、社区数据）均与该账号强绑定。用户可以直接使用手机号登录注册；也可以使用微信、微博授权登录注册，但注册后，仍然必须绑定手机号。
  • 实名认证（可选）：为防范黄牛刷单、批量恶意打卡及满足后续可能的高级权益兑换（如演唱会门票抽选、实物奖品邮寄），系统预留实名认证功能。用户在参与特定稀有活动前，需完成实名信息（姓名、身份证号）的绑定。
  • 账号安全与注销：支持用户修改登录密码、绑定/换绑第三方社交账号。提供账号注销功能，注销后账号绑定的所有数字资产将被永久清空且不可恢复，注销流程需设置二次确认与验证码校验机制。

5. 核心功能架构

App 主要分为以下几个核心模块：

  • 登录/注册模块：支持验证码、密码登录，以及首次登录的资料完善。
  • 首页模块：展示用户的数字专辑柜/收藏，支持添加新的收藏品。
  • 识别添加模块：通过 NFC 感应或扫描二维码绑定收藏品或者小卡、徽章等。
  • 专辑详情模块：数字专辑的核心消费场景，包含音乐播放、MV、写真、数字小卡、徽章展示及专属权益和社区。
  • 消息通知模块：接收系统通知与社区互动消息。
  • 个人中心模块：管理个人资料、账号设置、问题反馈等。
  • 客服模块：提供常见问题解答与人工在线客服支持。

7. 后续规划与建议

  • 数据埋点：建议在 NFC 识别成功率、核心页面停留时长、播放器使用率上做好数据埋点。
  • 性能与并发预估：
    • 大文件加载（音频/MV/写真）：由于数字资产包含大量无损音频、高清视频（MV）及高分辨率电子写真，预计单用户单次消费产生的下行带宽峰值较高。需引入阿里云 CDN 加速节点分发，依托其海量全网边缘节点与超大带宽储备，理论上可承载百万级用户同时在线播放无损音频或观看高清 MV，有效应对明星发售新专等极端峰值场景。同时，在 App 端需做好合理的分级缓存策略（如音乐文件的边播边存、图片缩略图预加载等），以保障播放流畅度并避免占用用户过多本地存储空间。
    • NFC 感应识别并发：在演唱会、线下快闪店等大型“公用 NFC”打卡场景中，预计会在极短时间内（如开场前后 1 小时内）产生瞬时高并发的识别打卡与资产下发请求。系统需具备弹性扩容能力，接口层应支持高并发处理，并通过消息队列等机制削峰填谷，确保现场用户“秒级响应”的打卡体验。
